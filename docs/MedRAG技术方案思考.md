# MedRAG：基于 MinerU 的医疗文献高质量 RAG 知识库

> **天机运算团队** · MinerU 赛道三（医疗赛题）  
> *"在 0 和 1 的海洋里，探索技术奥秘的工程师团队"*

---

## 摘要

医疗文献是临床证据的核心载体。然而，从一篇原始 PDF 到可供检索的高质量知识条目，中间横亘着一道鸿沟——复杂版式的解析偏差、固定字数切分导致的语义断裂、以及人工处理难以规模化。这三重困境使得医疗 RAG 知识库长期停留在"勉强可用"的水平，距"临床可靠"仍有显著差距。

MedRAG 是我们对这一鸿沟的系统性回答。它以 MinerU 2.5-Pro 作为解析核心，以循证医学的 PICO 框架为语义切分逻辑，以 FAISS 向量检索与 LightRAG 知识图谱为双引擎，以 Function Calling 驱动的 Agent 为推理中枢，构建了一套从 PDF 到智能问答的端到端自动化系统。

本文档将按照赛题要求的三个维度——**复杂版面解析**、**语义智能切分**、**端到端自动化构建**——逐一阐述我们的技术方案、关键决策、工程实践与创新突破。

## 一、赛题理解：我们要解决的不是一个技术问题，而是一个工程体系问题

赛题明确指出了三个核心痛点。在展开技术方案之前，有必要阐明我们对这三个痛点的理解深度——因为理解的层次决定了方案的高下。

### 痛点一：复杂版式及语义级解析困难

表面上看，这是"PDF 解析不准"的问题。但深入分析会发现，它实际上是两个子问题的叠加：

**（1）版式还原问题**：双栏/多栏排版、嵌套表格、图文混排——传统基于规则引擎的解析器（如 PyMuPDF、Docling 的布局分析模块）依赖坐标检测和启发式规则，在拉丁语系简单版式上可接受，但面对中文医学文献的复杂版面时，规则失效。

**（2）语义理解问题**：即使版式被正确还原，通用工具也不理解"基线表"和"结局表"的区别，不知道森林图中的 HR 值和 95% CI 意味着什么，无法识别图注与正文的引用关系。这是"解析"和"理解"的本质差距。

解决这个痛点，需要一个既能"看清版面"又能"读懂内容"的方案。纯规则引擎做不到语义理解，纯 OCR 做不到版式还原。视觉语言模型（VLM）是目前能同时兼顾两者的技术路径。

### 痛点二：语义级切片准确性低

固定字数切分的问题不在于"切分"，而在于"不懂"。512 字的滑动窗口可以把一篇文献切成几百块，但它不知道切在哪里——可能在 Methods 的第一句话中间，可能在 Primary Outcome 的关键数据前，也可能把基线表和它的解读一分为二。

这种碎片化直接导致 RAG 检索时返回的文本块缺乏完整的上下文。LLM 看到碎片后试图"脑补"缺失的信息——这就是**幻觉的核心诱因**。

好的切分不是"切得更细"，而是"切在正确的地方"——切分的粒度由内容的语义单元决定，而非预设的数字。

### 痛点三：人工处理效率极低

单篇文献数小时的标准化处理时间，根因不在于"人工速度慢"，而在于**缺乏工程化的系统设计**。解析、切分、提取、索引、检索——这些环节如果各自独立运行，每一步都需要人工衔接和校对。

解决效率问题，关键不是"让人更快"，而是让机器完成端到端的自动化——人只在最后审核结果。

## 二、任务要求一：复杂版面的精准识别与语义级解析

> *"参赛方案需能够准确解析双栏/多栏、图文混排的医疗文献，精准提取基线表、结局指标表、临床图表及正文内容，解决章节切分混乱和表格结构丢失等问题。"*

### 2.1 方案选择：为什么是 VLM 而不是规则引擎

我们在项目早期对比了三种解析路径：

| 路径 | 代表方案 | 版式还原 | 语义理解 | 中文支持 | 部署成本 |
|------|---------|---------|---------|---------|---------|
| 规则引擎 | Docling 布局分析 | 中（拉丁语系好） | 无 | 弱 | 低 |
| 远程 VLM | 云端 API 调用 | 高 | 高 | 取决于模型 | 按量付费 |
| 本地 VLM | Qwen2-VL 本地推理 | 高 | 高 | 原生中文 | 需 GPU |

我们最终选择了**本地 VLM（MinerU 2.5-Pro + Qwen2-VL 1.2B）**作为主解析引擎。核心理由：

**（1）视觉理解替代坐标规则。** Qwen2-VL 逐页渲染 PDF 为图像，直接理解页面上的文字、表格和布局关系——就像人类阅读一样。双栏不是两个互相干扰的坐标带，而是模型自然理解的视觉结构。在 4 篇测试文献中，双栏/多栏版式的章节归属准确率超过 90%。

**（2）表格语义保留。** VLM 不仅提取表格中的文字，还保留行列的逻辑关系。基线表的每一行仍然对应一个患者特征，结局表的每一列仍然对应一个终点指标。这为后续的图表语义分析（见 2.3 节）奠定了基础。

**（3）本地部署保障数据安全。** 医学文献可能包含患者隐私信息。本地推理确保数据不离开服务器，满足医疗场景的合规要求。RTX 5060（8GB VRAM）即可运行。

Docling 未被弃用，而是被保留用于图片提取——利用其成熟的 PDF 图片抽取能力，补充 VLM 在精确图片定位方面的不足。这是"各取所长"的工程思维，而非"非此即彼"的技术选型。

### 2.2 VLM 幻觉的检测与过滤

VLM 在解析超大或超复杂表格时，偶尔会陷入生成循环——反复输出同一个词数百次（例如 "Normal Normal Normal..."）。这种幻觉如果不加过滤，会污染整个知识库。

我们在**两个层级**部署了检测机制：

**层级一：解析阶段实时拦截。** 在 `mineru25pro_parser.py` 中，每完成一个页面的解析后，立即检测文本块中是否存在单字频次异常（某个词出现 >30 次且占总词数 >30%）。触发该阈值 → 丢弃该块。

**层级二：入库阶段质量门禁。** 在 FAISS 编码前，对每个文本块进行三项检查：长度 < 30 字符 → 丢弃；非字母数字占比 > 50% → 丢弃；单字重复 > 30% → 丢弃。三道防线确保进入知识库的每一条数据都经过质量验证。

**实际效果**：在 4 篇文献的 80 个解析块中，检测到 4 个幻觉块（5%），全部被拦截，零漏网。

### 2.3 图表语义理解：从像素到结构化数据

赛题特别强调了对"基线表、结局指标表、临床图表"的语义解析能力。这恰恰是传统 PDF 工具最薄弱的环节。

我们的方案构建了一条完整的图表语义理解链路：

```
PDF 图片 → Docling 提取 → Moonshot Vision 分类（5 型）
    ├─ baseline_table  → 基线特征表 → LLM 序列化为自然语言
    ├─ outcome_table   → 结局指标表 → 效应量 + CI + p 值提取
    ├─ forest_plot     → 森林图 → HR/RR + 异质性 I² 提取
    ├─ km_curve        → KM 曲线 → 中位生存期 + log-rank p 值
    └─ flowchart       → 流程图 → 流程结构描述
```

关键创新在于：我们不满足于"提取图片中的文字"，而是让 VLM 对图表进行**类型识别 + 结构化数据提取**。例如，当模型判定一张图为"森林图"后，会使用专门针对森林图的 prompt 来提取 HR 值、置信区间和异质性指标——而不是用通用 prompt 做泛化的图片描述。

这一设计使得图表从"不可检索的图片"变成了"可查询的结构化数据"。在 seyfarth2008 文献中，系统成功提取了包含 6 张图表的完整数据，7 张提取图表全部正确关联到对应的文本段落。

### 2.4 解析格式与评测对齐

我们的解析输出格式对齐 MinerU JSON 规范，保持了 page_idx、bounding_box、content_type 等元数据字段的完整性。同时，文本块的质量指标（证据等级、PICO 分类置信度、幻觉过滤状态）为 Omnidocbench 评测方法提供了可直接验证的数据支撑。

## 三、任务要求二：基于语义的智能文本切分

> *"摒弃传统的固定字数拆分方式，实现基于章节逻辑和语义完整性的智能文本块切分（例如 Primary outcome、Secondary outcome、Subgroup analysis、Sensitivity analysis 等），确保切分后的文本块在 RAG 检索中能提供完整且连贯的上下文支撑，同时检索来源可追溯、多文献证据一致性判断等。"*

### 3.1 从固定字数到语义单元

固定字数切分等价于"闭着眼睛切菜"——不看内容，只数数字数。对于小说这种线性叙事文本，这或许勉强可行。但对于医学文献这种有严格逻辑结构的信息密集型文本，这就是灾难——完整的临床推理链被从中切断，数据的展示和解读被分到两个块中，检索时检索到的是残缺的语义。

我们的方案从根本上改变了切分的逻辑：**不是"按字数切"，而是"按语义单元分"**。

### 3.2 PICO 11 类医学语义分型

基于循证医学的 PICO 框架（Population / Intervention / Comparison / Outcome），我们定义了 11 种医学语义类型：

| 分类 | 英文 | 说明 | 切分逻辑 |
|------|------|------|---------|
| 主要结局 | primary_outcome | 研究的一级终点指标 | 独立成块，完整保留所有终点数据 |
| 次要结局 | secondary_outcome | 研究的二级终点指标 | 独立成块，与主要结局分离以支持独立检索 |
| 亚组分析 | subgroup_analysis | 按人群/特征分层分析 | 每个亚组独立成块 |
| 安全性结局 | safety_outcome | 不良事件/AE 报告 | 独立成块，支持安全性专项检索 |
| 方法设计 | methods_design | 研究设计/纳排标准 | 保留完整的方法学描述 |
| 基线特征表 | baseline_table | 入组人群基线数据 | 表格数据独立保留，附带解读文本 |
| 图表描述 | figure_description | 图表标题与说明 | 图表与对应文本联动 |
| 讨论 | discussion | 结果解读与临床意义 | 保留完整的推理链 |
| 结论 | conclusion | 研究总结与建议 | 精炼提取核心结论 |
| 参考文献 | references | 引用列表 | 保留引用网络用于溯源 |
| 其他 | other | 未分类的补充内容 | 保留文档完整性 |

### 3.3 实现路径：规则 + AI 双层策略

分类不是靠硬匹配关键词。我们的实现路径分为三步：

**第一步：段落级初始拆分。** 基于双换行和章节标题识别，将文本拆分为 250-1200 字符的初始段落。过短的自动向前合并，过长的按句号边界再拆分。

**第二步：PICO 批量分类。** 将所有初始段落通过 DeepSeek API 进行批量的语义分类。模型理解内容后判断每个段落属于哪种 PICO 类型。这一步利用了 LLM 的语义理解能力，避免了关键词匹配的脆弱性。

**第三步：全局语义合并。** 相邻段落即使 PICO 类型不同，如果语义连续，也应该合并。例如 Results 的最后一段和 Discussion 的第一段如果讨论同一个发现，合并为一个块可以保留完整的推理上下文。这一步通过 DeepSeek-Chat 判断相邻对的语义连续性，并在置信度足够时执行合并。

**实际效果**：在 4 篇文献的 60 个文本段中，PICO 分类的准确率达到 85%+（以人工标注为基准）。平均块大小为 680 字符，标准差 210 字符——既不过大（防止检索噪音），也不过小（防止语义碎片化）。

### 3.4 检索可追溯性与多文献证据一致性

赛题明确要求"检索来源可追溯"和"多文献证据一致性判断"。这两点与语义切分直接相关：

**可追溯性**：每个文本块在入库时绑定了 `doc_name`、`page_idx`、`evidence_level`、`section` 四维元数据。Agent 返回答案时，每条引用精确到文献名 + 页码 + 证据等级，评审者可以逐一验证。

**一致性判断**：Agent 的 `cross_check` 工具通过检索同一主题下不同文献的结论片段，对比其 PICO 分类和证据等级，检测结论矛盾。当多篇文献的结论不一致时，Agent 会明确标注矛盾并说明各方的证据质量——而非简单地"综合"出一个调和的结果。

## 四、任务要求三：端到端知识库自动化构建

> *"实现从原始医疗 PDF 文献输入，到高质量 RAG 知识库输出的端到端自动化处理，大幅缩减人工校对时间，支撑医疗知识库的规模化构建。"*

### 4.1 全链路自动化流水线

我们的自动化流水线分为 5 个阶段，由 MySQL 状态机精确管控：

```
PDF 上传 → MinerU 2.5-Pro VLM 逐页解析 (Qwen2-VL 1.2B)
    │
    ├─ 段落拆分 + VLM 幻觉过滤
    ├─ PICO 批量语义分类 (DeepSeek API)
    ├─ 图表提取 + 结构化分析 (Docling + Moonshot Vision)
    │
    ▼
FAISS 向量索引 (BGE-M3 1024-dim)
    │
    ├─ 质量门禁 (3 项检查)
    ├─ 增量索引 (MD5 精确差分)
    ├─ 失败自动快照回滚
    │
    ▼
MySQL 业务同步
    │
    ├─ articles 表 (文献元数据)
    ├─ text_segments 表 (PICO 语义段落)
    └─ extracted_figures 表 (结构化图表)
    │
    ▼
LightRAG 知识图谱构建
    │
    ├─ 实体-关系联合抽取 (DeepSeek API)
    ├─ 节点分类 (9 种医学实体类型)
    └─ 图谱持久化 (NetworkX + JSON)
    │
    ▼
可检索 / 可问答
```

**实测数据**：单篇 2 页文献（含 2 张表格），从上传到可检索，全程约 6 分钟（RTX 5060）。其中 VLM 解析占比约 73 秒（72 秒/页），FAISS 编码约 7 秒，LightRAG 图谱构建约 30 秒。人工参与时间为零。

### 4.2 全流程状态机管理

每个上传任务在 MySQL `upload_tasks` 表中拥有完整的生命周期追踪：

| 状态 | 含义 | 记录的关键指标 |
|------|------|-------------|
| received | 文件已接收，等待处理 | 文件 MD5、大小、上传时间 |
| parsing | MinerU 2.5-Pro 解析中 | 解析耗时（ms）、VLM 引擎、幻觉过滤数 |
| chunking | FAISS 向量编码 + MySQL 同步 | 向量增量、编码设备（GPU/CPU） |
| indexing | LightRAG 知识图谱构建 | 实体数、关系数、构建耗时 |
| done | 全流程成功 | 各阶段累计耗时 |
| partial | 部分成功（FAISS 可用，图谱不可用） | 失败阶段及错误详情 |
| failed | 处理失败 | 失败阶段及错误详情 |

每一步都有**独立超时保护**（解析 600s、FAISS 300s、LightRAG 300s），每一步的状态变更都有精确的时长记录，每一步的失败都有明确的根因标注。这不是花架子——它让运维者可以在任何时刻精准定位问题。

### 4.3 MySQL + FAISS + LightRAG 三系统一致性

三个独立存储系统的数据必须一致，否则查询结果会自相矛盾。我们在每次服务启动时执行交叉校验：MySQL 的文章数 == FAISS 的文档数；MySQL 的文本段数在 FAISS 向量数的合理倍数范围内；不存在 MySQL 中有记录但 FAISS 中无对应向量的孤行。当前系统状态：4 篇文章在三个系统中完全同步，零不一致。

## 五、核心技术突破：Agent 多跳推理引擎

赛题要求在"精准溯源"基础上实现"医疗问答的专业性"。我们认为，专业性不仅体现在答案的准确性上，更体现在**推理过程的透明和可验证**上。

### 5.1 从单次检索到多步推理

绝大多数 RAG 系统的模式是：用户提问 → FAISS 检索一次 → 把前 5 个结果塞给 LLM → 生成回答。这个模式对于简单的事实查询（"什么是房颤"）有效，但对于复杂的临床问题（"对比 Impella LP2.5 与 IABP 在心源性休克中的疗效差异并评估证据质量"）完全不够——它需要跨文献对比、证据质量评级、一致性判断等多步操作。

我们构建了一个 **Function Calling 驱动的多步推理 Agent**，拥有 9 个专业工具：

| 工具 | 能力 |
|------|------|
| search_rag | FAISS 向量检索 + LightRAG 图谱查询（双路协同） |
| deep_retrieve | 多临床维度并发检索 |
| cross_check | 多文献一致性交叉验证 |
| get_evidence | 单文献覆盖范围查询 |
| list_docs | 知识库全文献列表 |
| extract_chart | 图表相关文本提取 |
| analyze_image | VLM 图表结构化数据分析 |
| estimate_grade | GRADE 循证证据质量评级 |
| build_consistency_matrix | 多文献结论一致性矩阵 |

Agent 的推理流程是：

1. **意图分类**：5 类（事实查询 / 证据综合 / 图表分析 / 多文献对比 / 系统综述）
2. **自主工具调用**：LLM 决定调用哪些工具、以什么顺序调用
3. **回溯自纠**：置信度低时，自动简化检索词重新搜索（≤2 次）
4. **搜停控制**：5 次检索后强制生成回答，防止过度检索
5. **引用溯源**：每条结论绑定 `[文献名, 页码, 证据等级]`

**实测验证**：对于一个跨 3 篇文献的临床对比问题，Agent 执行了 5 步推理（list_docs → search_rag ×3 → 最终回答），耗时 57 秒，返回了包含 8 条可追溯引用的结构化回答。评审者可以通过引用的 `[doc, page, evidence_level]` 三元组逐一验证每一条结论。

### 5.2 GRADE 证据评级与一致性矩阵

这两个工具是我们对赛题"多文献证据一致性判断"要求的直接回应。

**GRADE 评级**：系统自动对检索到的证据进行 GRADE 证据质量评级（高/中/低/极低），标注降级因素（偏倚风险、不一致性、间接性、不精确性、发表偏倚）和推荐强度。这使得 Agent 在回答时能够区分"基于高质量 RCT 证据的强推荐"和"基于病例系列的弱推荐"。

**一致性矩阵**：当多篇文献就同一主题得出不同结论时，`build_consistency_matrix` 构建一个 N×N 的一致性判定矩阵，标注每对文献的一致性状态（一致/矛盾/互补/正交）和判定依据。Agent 在回答中会明确标注证据矛盾——而非粉饰出一致性。

### 5.3 四层容错体系

在医疗领域，系统的可靠性比回答的完美性更重要。我们构建了四层容错：

1. **连接重试**：LLM 调用失败 → 3 次重试（间隔 2s/4s）
2. **模型降级**：主 LLM 全部失败 → 自动切换备用 LLM
3. **FAISS 直接检索**：所有 LLM 都不可用 → 不经过 LLM 推理，直接返回原始检索片段
4. **索引快照回滚**：FAISS 写入失败 → 自动恢复到写入前的快照状态

每一层降级都记录在审计日志中，运维者可以追踪每一次降级的原因和时间。

## 七、创新与突破点

### 7.1 技术创新

1. **VLM 替代规则引擎用于医学 PDF 解析**：这是本项目的核心创新。我们在医疗文献这一垂直场景中验证了"视觉语言模型逐页理解"替代"规则引擎坐标匹配"的可行性，并在版式还原准确率和表格语义保留两个维度上取得了优于纯规则方案的实测结果。

2. **PICO 11 类医学语义分型 + LLM 全局合并**：将循证医学框架引入文本切分逻辑，实现了从"固定字数切"到"按语义单元分"的范式转变。LLM 判断相邻段语义连续性的做法，比传统滑动窗口的重叠策略更加精准。

3. **双引擎 RAG 架构的互补设计**：明确界定了 FAISS（相似文本检索）和 LightRAG（概念关系查询）的分工，避免了两种技术的功能重叠和资源浪费。"文献证据"和"知识背景"的分层引用规则，在学术界和工业界 RAG 实践中均属较为前沿的设计。

4. **Agent 多跳推理 + 回溯自纠**：将 Function Calling 应用于医学 RAG 场景，实现了从"一次检索 + 一次生成"到"自主规划检索策略 + 多步推理 + 自我纠错"的升级。

### 7.2 工程创新

1. **MySQL 状态机全流程追踪**：每个上传任务的每个阶段都有精确的时长指标和失败根因，运维者可以在秒级定位问题。

2. **MD5 精确差分增量更新**：文献版本变更时，仅对 MD5 哈希发生变化的文本块执行 FAISS 删除和重新编码，精度 100%。

3. **四层容错体系**：从连接重试到模型降级到 FAISS 直检到快照回滚，形成了完整的故障处置链路。

4. **结构化审计日志**：每次 API 调用、每个管道事件都生成 JSON 格式的审计日志，配合自动化审计脚本实现一键全链路验证。

### 7.3 方法创新

1. **业务驱动的技术选型**：每一个技术选择都由医疗业务需求驱动——PICO 分类来自循证医学框架，GRADE 评级来自临床证据评估体系，图表 5 型分类来自医学文献图表类型学。

2. **语义理解优先的架构哲学**：在解析、切分、检索、推理四个核心环节中，一致性地选择了"让模型理解内容"而非"写规则匹配模式"的路径。这种一致性降低了系统的脆弱性和维护成本。

## 八、应用场景与价值

### 8.1 临床科研

研究员上传一篇新的 Meta 分析 PDF，系统自动完成解析、切分、入库。3 分钟后，可以通过自然语言提问："这篇 Meta 分析纳入的 RCT 中，实验组和对照组的样本量分布如何？"系统返回精确数据 + 原文页码引用。原本需要数小时的手工提取、整理、标注工作，压缩为零。

### 8.2 医疗大模型研发

RAG 训练数据质量直接决定模型回答质量。MedRAG 提供按 PICO 类型标注的语义段落，每条段落绑定文献名、页码和证据等级，从根本上减少了 RAG 数据中的语义碎片化问题——这是医疗大模型幻觉的核心贡献因子。

### 8.3 临床决策支持

临床医生在面对复杂病例时，可以通过 Agent 查询最新循证证据："对于 CHA₂DS₂-VASc≥2 的房颤患者，DOAC 和华法林在颅内出血风险方面有何差异？"Agent 自动检索知识库、对比多篇文献结论、输出 GRADE 评级和一致性判断——所有结论都可追溯到原文。

### 8.4 MedBench 评测对齐

我们的系统架构与 MedBench 评测体系高度对齐：
- 文本块的质量指标（证据等级、PICO 分类置信度）可直接用于 Macro-Recall 和 BERT-Score 评估
- 引用溯源机制（doc/page/evidence_level）为答案准确性验证提供了完整的审计链路
- Agent 的推理过程记录（每一步的工具调用、参数和输出）为 LLM-as-a-Judge 评估提供了透明且可复现的评判依据
