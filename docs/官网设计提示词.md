**标题**：医疗知识数字化，为什么这么难？

**三列卡片布局**，每列一张痛点卡片（hover 时卡片微微上浮 + 阴影加深）：

| 卡片 1 | 卡片 2 | 卡片 3 |
|--------|--------|--------|
| **🔬 复杂版式解析困难** | **✂️ 语义切分碎片化** | **⏱️ 人工处理效率极低** |
| 双栏/多栏混排、医学表格图表结构丢失，传统 PDF 工具版式还原准确率极低 | 固定字数拆分破坏章节逻辑，文本块语义不完整，是医疗大模型产生幻觉的核心诱因 | 单篇高质量文献处理需数小时，缺乏工程化系统，无法规模化构建 |
| 现有工具：PyMuPDF、Docling 对中文医疗文献处理效果差 | 现有方式：按 512 字硬切，PICO 信息被截断 | 现有流程：科研人员手动拆分、校对、标注 |

**底部小结文字**：
"这三个问题环环相扣，导致医疗 RAG 知识库长期停留在 '能用但不好用' 的阶段。我们决定从头解决。"

### 字体系统
```
标题字体：'Inter' 或 '思源黑体' — Bold (700-800)
正文字体：'Inter' 或 '思源黑体' — Regular (400)
代码字体：'JetBrains Mono' 或 'Fira Code' — Regular (400)
```

### 间距系统
```
区块上下间距：120px (桌面) / 80px (移动)
区块左右内边距：max-width 1280px 居中
卡片间距：24px
内容内边距：32px
```

### 动效规范
```
过渡时长：200-400ms
缓动函数：cubic-bezier(0.16, 1, 0.3, 1)
hover 效果：translateY(-4px) + box-shadow 加深
滚动触发：Intersection Observer, 元素从下向上淡入 (translateY(30px) → 0, opacity 0 → 1)
视差效果：Hero 背景层移动速度为滚动的 0.3x
```

### 响应式断点
```
Mobile:  < 768px  — 单列布局，卡片全宽
Tablet:  768-1024px — 两列布局
Desktop: > 1024px — 全布局（三/四列）
```

## 六、技术实现提示

### 性能目标
- Lighthouse Performance Score ≥ 90
- First Contentful Paint < 1.5s
- Largest Contentful Paint < 2.5s
- 页面总大小 < 2MB（含图片）
- 所有图片使用 WebP 格式 + 懒加载

### SEO 要求
- 每个区块使用语义化 HTML5 标签（`<section>`, `<article>`, `<header>`, `<footer>`）
- 所有图片包含 `alt` 属性
- `<title>`: MedRAG — 医疗文献高质量 RAG 知识库 | 天机运算
- `<meta description>`: MedRAG 是基于 MinerU 2.5-Pro 的端到端医疗文献 RAG 知识库系统，实现从 PDF 到智能问答的全链路 AI 驱动自动化。由天机运算团队开发。
- Open Graph 标签（用于社交媒体分享预览）

### 兼容性
- Chrome 90+, Firefox 90+, Safari 15+, Edge 90+
- 移动端完整适配（不缩功能，只调布局）
