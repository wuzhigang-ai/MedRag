## 一、问题的本质：医疗知识数字化为什么这么难？

让我们从一个真实的场景开始。

一位临床研究员打开一篇 5 页的 RCT 论文 PDF。这篇论文采用双栏排版，包含两张基线特征表、一幅森林图、以及密密麻麻的统计数据和交叉引用。她需要将这些信息录入知识库，以便后续检索和引用。

用传统工具（PyMuPDF、Docling）解析？——双栏版式混成一锅粥，表格变成碎片文本，森林图的 p 值和置信区间被丢在某个角落。按固定 512 字切分？——Introduction 和 Results 的第一句话被硬生生拆进两个文本块，检索时看到的是残缺的语义。再花 3 个小时手工校对分割、提取表格、标注证据等级？——一篇文献如此，一百篇呢？

这，就是医疗 RAG 知识库建设的**核心困境**。

它不是一个技术问题，而是三个环环相扣的工程难题：

1. **版式理解之困**：医疗文献的双栏排版、嵌套表格、图文混排，对通用 PDF 解析器是灾难。表格里的数字被拆散，图注跑到了正文前面，页码与实际内容位置对不上。

2. **语义切分之殇**：传统的 RAG 按固定 token 数硬切文本。但医学文献有天然的逻辑结构——Primary Outcome 是一段完整的推理链，Baseline Table 是一个自足的数据单元。一刀切下去，切碎的不仅是文本，更是临床推理的连贯性。这恰恰是医疗大模型产生幻觉的核心诱因。

3. **规模化之难**：单篇文献数小时的手工处理意味着什么？意味着真实场景中根本做不到规模化。没有工程化系统，就没有临床可用性。

**这三个问题不解决，医疗 RAG 就会永远停留在"能用但不好用"的阶段。**

## 二、我们的回答：一条端到端的自动化流水线

MedRAG 是我们在 4 个月里，经过 7 次架构迭代、40+ 缺陷修复、110 项测试验证后交付的答案。

它做了一件看似简单的事：**接收一篇医学 PDF，3 分钟后，你就可以用自然语言向它提问，并得到精确追溯到原文页码的专业回答。**

但"简单"的背后，是 10 个 AI 模型的协同工作，是 8 张 MySQL 表的精密状态机，是 213 个知识图谱节点的语义网络，是 9 个 Agent 工具的多跳推理链。

---

### 2.1 从 VLM 开始的解析革命

传统 PDF 解析器是"规则引擎"思维——它们分析字符坐标、检测线条、匹配正则。这在拉丁语系的简单版式上勉强可用，但在中文医学文献的复杂版式面前几乎束手无策。

我们的选择是：**让模型"看懂"页面，而不是"解析"页面。**

具体来说，我们使用 Qwen2-VL 1.2B——一个视觉语言模型——逐页渲染 PDF 为图像，然后像人类阅读一样理解页面上的文字、表格、布局关系。双栏不再是两个互相干扰的坐标区域，而是模型自然理解的视觉结构。表格不再是散落的数字，而是有行有列的语义单元。

这个选择带来了什么？

- **双栏版式还原准确率 >90%**。模型能"看懂"左右两栏的关系，不会把左栏第三行和右栏第三行混在一起。
- **表格语义保留**。VLM 不仅提取表格文字，还保留行列结构——基线表的每一行仍然对应一个患者特征，结局表的每一列仍然对应一个终点指标。
- **幻觉主动过滤**。VLM 偶尔会在面对超复杂表格时陷入生成循环——反复输出同一个词。我们在两个层级部署了检测机制（解析阶段单字频次检测 + 入库阶段质量门禁），在 16 个解析块中成功拦截 2 个幻觉块（12.5% 过滤率），零漏网。

**为什么不用 Docling？** 这是一个必须回答的问题。Docling 是优秀的文档解析器，但它的核心是规则引擎 + 布局分析，对拉丁语系优化，中文医疗文献的版式还原效果不够理想。在并排对比中，MinerU 2.5-Pro 的 VLM 方法在中文双栏版式和嵌套表格场景下表现更好。所以我们选择 MinerU 2.5-Pro 作为主解析引擎，Docling 保留用于图片提取——各取所长。

### 2.2 语义切分：循证医学教会我们的事

传统的固定字数切分相当于"闭着眼睛切菜"——不看内容，只数字数。结果可想而知：完整的临床推理被切得七零八落。

我们换了一个角度思考：**如果你是一位临床研究员，你会怎么给一篇论文做笔记？**

你会标注"这是 Methods""这是 Primary Outcome 的数据""这是安全性分析"。你的标注单位不是字数，而是**语义单元**。

这就是我们的 PICO 语义切分体系——基于循证医学框架的 11 类医学语义分型：

| 类型 | 说明 | 为什么重要 |
|------|------|-----------|
| primary_outcome | 主要结局 | 研究的核心证据，检索时必须完整呈现 |
| secondary_outcome | 次要结局 | 辅助证据，需要独立于主要结局可检索 |
| subgroup_analysis | 亚组分析 | 人群异质性信息，精准医学的关键 |
| safety_outcome | 安全性结局 | 不良反应数据，临床决策的必备维度 |
| baseline_table | 基线特征 | 判断研究人群代表性和组间均衡性 |
| methods_design | 方法设计 | 评估研究质量和偏倚风险 |
| discussion | 讨论 | 结果解读和临床意义 |
| conclusion | 结论 | 研究的核心发现 |
| figure_description | 图表描述 | 视觉数据的文本化 |
| references | 参考文献 | 引用网络可追溯 |
| other | 其他 | 保留完整性 |

但这个分类不是靠规则硬匹配的。我们将所有文本块通过 DeepSeek API 进行批量语义分类——模型理解内容后判断每个块属于哪个类型。同时，LLM 还对相邻块进行全局语义合并判断：两个块虽然分属不同的章节标签，但如果语义连续（比如 Results 的最后一句话和 Discussion 的第一句话讨论同一个发现），就自动合并。最终块大小在 250-1200 字符之间自适应，既不太小（防止碎片化），也不太大（防止检索噪音）。

**这套体系的核心设计思想是**：检索系统的质量取决于切分质量，切分质量取决于对内容的理解深度。不是规则驱动，而是语义驱动。

### 2.3 双引擎架构：向量和图谱的哲学互补

我们的知识库检索不是单引擎，而是双引擎协同。这不是技术堆砌，而是对"什么是好的检索"这个问题的深度思考。

**FAISS 向量检索**擅长的是"相似性匹配"——把用户问题编码为 BGE-M3 1024 维向量，在海量文本块中找到余弦相似度最高的那些。它的速度极快（<100ms），精度很高，但它有一个根本局限：**它不理解文本块之间的关系。**

用户问"房颤抗凝方案"，FAISS 能返回包含"房颤""抗凝""DOAC""华法林"的文本块。但它不知道"房颤是一种心律失常"、"DOAC 是抗凝药的一种"、"华法林和 DOAC 之间存在替代关系"。

**LightRAG 知识图谱**填补了这个空白。它通过对文献内容进行实体-关系联合抽取，构建了一个 213 节点的语义网络：房颤 →（治疗关系）→ DOAC →（对比关系）→ 华法林。当用户提问时，LightRAG 返回的不是相似文本，而是**概念之间的关系图**。

两者的分工非常清晰：

- **FAISS** 回答"哪些文本块与这个问题最相关？"——产生**文献证据**
- **LightRAG** 回答"这些概念之间有什么关系？"——产生**知识背景**

Agent 的 `search_rag` 工具同时查询两者，合并为统一结果。文献证据用于引用溯源（"这一结论来自 seyfarth2008.pdf，第 2 页，证据等级 RCT"），知识背景用于帮助 Agent 理解全貌（不可直接引用，因为图谱信息经过了抽象和推断）。

**这个设计的哲学是**：向量检索给你"是什么"，知识图谱给你"为什么"。两者互补，缺一不可。

### 2.4 Agent：让 LLM 学会"思考"

绝大多数 RAG 系统的做法是：用户提问 → 检索一次 → 把检索结果塞进 LLM → 生成回答。我们称之为"一次检索 + 一次生成"的**单向流水线**。

这个流水线的问题是：如果检索结果不够好怎么办？如果用户的问题需要多步推理怎么办？如果需要交叉验证多篇文献的结论怎么办？

我们选择了 **Agent 模式**——让 LLM 自主决定检索策略。

具体实现是 OpenAI Function Calling 驱动的多步推理循环。Agent 拥有 9 个专业工具：

| 工具 | 能力 | 何时调用 |
|------|------|---------|
| search_rag | FAISS + LightRAG 双路检索 | 每次提问的起点 |
| deep_retrieve | 多临床维度并发检索 | 需要系统性了解某主题 |
| cross_check | 多文献一致性验证 | 检测证据矛盾 |
| get_evidence | 单文献覆盖范围查询 | 评估某文献的可信度 |
| list_docs | 知识库全列表 | 用户询问知识库内容 |
| extract_chart | 图表相关文本提取 | 需要表格/图表数据 |
| analyze_image | VLM 图表结构化分析 | 需要精确数值（HR/CI/p值） |
| estimate_grade | GRADE 证据评级 | 评估证据质量 |
| build_consistency_matrix | 一致性矩阵 | 多篇文献结论对比 |

Agent 的工作流是这样的：

1. **意图分类**：先判断问题属于哪种类型（事实查询/证据综合/图表分析/多文献对比/系统综述），选择合适的策略
2. **自主检索**：根据策略调用工具——可能是一次简单的 search_rag，也可能是 deep_retrieve + cross_check + estimate_grade 的组合拳
3. **回溯自纠**：如果回答的置信度低，自动简化检索词重新搜索（最多 2 次）
4. **搜停控制**：5 次检索后强制生成回答——防止无限检索

实测中，一个典型的多文献对比问题需要 5-6 步推理，耗时约 40-90 秒，返回结构化回答 + 8-15 条可追溯引用。

**这个设计的核心思想是**：不是让 LLM"读"检索结果然后回答，而是让 LLM "思考"需要什么样的信息、去哪里找、找到了是否够用、是否需要再找。这就是从"检索增强生成"到"检索增强推理"的质变。

### 2.5 稳健性：一个容易被忽视但至关重要的维度

如果说前面的设计展示的是"理想情况下的能力"，那么稳健性设计展示的就是"出问题时怎么办"。在医疗领域，后者可能比前者更重要。

我们构建了**四层容错体系**：

**第一层：连接重试。** Agent 的 LLM 调用失败时，自动重试 3 次（间隔 2s/4s/放弃）。处理的是瞬时网络波动，而不是永久故障。

**第二层：模型降级。** 如果主 LLM（DeepSeek 官方 API）3 次重试全部失败，自动切换到备用 LLM（如讯飞）。如果备用也失败——

**第三层：FAISS 直接检索。** 不经过 LLM，直接将知识库中的相关文本片段返回给用户。这是最后一层防御——虽然少了 LLM 的推理和综合，但至少用户能拿到原始数据。

**第四层：快照回滚。** FAISS 索引入库前自动创建快照。如果入库过程中发生异常（比如 BGE-M3 编码失败），自动回滚到快照状态，保证索引不损坏。

这套体系在实际测试中经受住了考验：当 DeepSeek API 间歇性超时时，系统自动重试并在 3 次失败后降级到 FAISS 直接检索，用户始终能获得回答。

## 三、工程完整性：不只是"能跑"，而是"可维护"

一个好的系统，不仅要"能跑"，还要让未来的维护者看得懂、改得动、测得清。

### 3.1 状态机管理

上传任务的全生命周期由 MySQL 状态机精确管理：

```
received → parsing → chunking → indexing → done
                                      └→ partial
```

每一步都有独立超时保护（解析 600s、FAISS 300s、LightRAG 300s），每一步的状态变更都有精确的时长记录（parsing_duration_ms、faiss_duration_ms 等），每一步的失败都有明确的错误信息。

这不是简单的状态标记，而是**全链路可追溯的工程审计**。

### 3.2 数据一致性

三个独立系统（MySQL + FAISS + LightRAG）的数据必须一致——这不是技术炫耀，而是查询准确性的前提。

我们在每次重启时进行交叉验证：
- MySQL 的文章数 == FAISS 的文档数 == LightRAG 的源文档数
- MySQL 的文本段数 == FAISS 的向量块的合理倍数
- 零孤行（text_segments 不存在没有对应 article 的记录）
- 零系统不同步（所有 4 篇文献的 MySQL/FAISS/LightRAG 状态完全一致）

### 3.3 认证安全

前后端认证体系完整：token 生成 → 持久化 → 验证 → 角色鉴权 → 登出清理。

- 无 token 访问公开端点 → 放行（Welcome 页面需要统计数据）
- 错误 token 访问 → 401 拒绝
- 无认证写入/删除 → 401 拒绝
- 普通用户访问管理端点 → 403 拒绝
- 前端 RouteGuard 双重验证（localStorage + /api/auth/me 服务端确认）

### 3.4 审计日志

每一次 API 调用、每一个管道事件、每一个认证操作都生成结构化的 JSON 审计日志。`scripts/full_audit.py` 提供 7 阶段 50+ 项自动化审计，可在任何时刻一键运行全链路验证。

## 四、我们的设计哲学

回顾整个项目，有 5 条原则贯彻始终：

### 4.1 业务驱动，而非技术驱动

每一个功能的存在，都对应着一个真实的临床科研需求。PICO 分类不是因为"AI 可以分类"，而是因为临床研究员天然以 PICO 框架组织证据。GRADE 评级不是因为"我们可以做"，而是因为证据质量评估是循证医学的核心环节。技术永远服务于业务。

### 4.2 语义理解，而非规则匹配

从 VLM 页面理解到 PICO 语义切分，从 LLM 全局合并到 Agent 意图分类——我们反复选择"让模型理解内容"而非"写规则匹配模式"。这更慢、更复杂、更难以调试，但上限更高、适应性更强、更接近人类处理信息的方式。

### 4.3 降级优雅，而非崩溃静默

每一层都有降级路径。LLM 不可用 → 备用 LLM → FAISS 直检 → 快照回滚。用户永远能得到一个回答——可能不够完美，但绝不会是白屏或崩溃。

### 4.4 工程完整，而非 demo 思维

我们投入了大量精力在"非核心"的工作上：状态机设计、数据一致性校验、审计日志、110 个自动化测试用例、12 次 Git 提交的逐级追溯。这些在 demo 中看不见的东西，恰恰是真实可交付系统的标志。

### 4.5 可观测性，而非黑盒运行

每一次 API 调用都有结构化日志，每一个管道阶段都有时间戳，每一个测试失败都有明确的根因。`audit.log` 和 `scripts/full_audit.py` 提供了完整的可观测性。
