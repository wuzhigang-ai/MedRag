## 一、产品宣言

**医学知识的价值，不在于被存储，而在于被使用。**

全世界每天有超过 4,000 篇医学论文发表。PubMed 收录的文献总量已突破 3,600 万篇。每一篇临床指南、每一项 Meta 分析、每一个 RCT 的结果，都可能是某个患者治疗方案的关键证据。

然而现实是：**绝大多数医学文献在 PDF 格式中沉睡。** 它们被下载、被存储、被遗忘——从未被真正检索、引用、对话。

为什么？因为从 PDF 到可检索的知识条目，中间横亘着一道巨大的工程鸿沟。这道鸿沟不是某个单一技术问题，而是一个系统性的工程难题——它涉及版面解析、语义理解、知识组织、推理能力、交互设计五个层次。任何一个层次的短板，都会让整个链条断裂。

**MedRAG 存在的唯一理由，就是填平这道鸿沟。**

## 二、用户画像：谁在为"不能检索的 PDF"付出代价？

### 2.1 临床研究员 —— 被困在手工劳动中的知识工作者

李医生是某三甲医院心内科的副主任医师。她正在主持一项关于"新型口服抗凝药在老年房颤患者中的安全性"的系统综述。为此，她需要筛选 247 篇候选文献，逐一阅读、标注、提取关键数据。

她的一天是这样度过的：打开一篇 PDF → 手动标注 Primary Outcome → 复制粘贴基线表数据到 Excel → 截图森林图 → 写摘要笔记 → 下一篇。一篇 5 页的 RCT 需要 45 分钟。247 篇就是 185 小时——相当于 23 个工作日。

**这不是智力劳动。这是体力劳动。** 李医生花了 8 年时间拿到博士学位，不是为了做数据录入员的。

### 2.2 医疗 AI 研发团队 —— 被脏数据拖累的算法工程师

张工是一家医疗大模型公司的 RAG 引擎负责人。他的团队面临一个尴尬的局面：微调后的模型在公开基准上表现优异，但在真实医院场景中却频繁产生幻觉。

根因分析指向同一个问题：**知识库中的文本块是固定字数切分的产物。** 一段完整的"亚组分析"被切成三块，检索时只能命中中间那一块——前半截的"HR=0.72"和后半截的"95% CI 0.58-0.89"都不在检索结果中。LLM 看到了不完整的数据，然后"脑补"了缺失的部分。

**这不是模型的问题。这是知识库质量的问题。** 垃圾进，垃圾出——这句计算机科学的古老箴言，在医疗 RAG 领域格外刺耳。

### 2.3 医院信息科 —— 被孤岛数据困住的决策者

王主任负责某省级医院的临床决策支持系统建设。他的服务器里躺着过去 5 年所有的院内指南和学术文献——超过 10,000 篇 PDF。

但这些 PDF 的现状是：**可以被搜索文件名，但无法被搜索内容。** 临床医生想要查"PD-1 抑制剂联合化疗在 NSCLC 脑转移亚组中的疗效"，得到的不是数据，而是一句"请自行查阅附件"。

**信息存在，但不可访问。这是数字时代的讽刺。**

## 三、产品定义：MedRAG 是什么？

MedRAG 是一个**端到端的医疗多模态 Agentic RAG 知识库系统**。这不是一句口号。让我们拆解每一个定语：

**端到端**：从 PDF 上传开始，到智能问答结束。中间没有"人工校对环节"、"手动标注步骤"、"请复制粘贴到这里"的提示。全程自动化，零人工干预。

**多模态**：不仅处理文本。基线表、森林图、KM 曲线——这些医学文献中最有价值的信息载体——被 VLM 识别、分类、结构化提取，成为可检索的数据，而非不可检索的像素。

**Agentic**：不是"用户问一句，系统检索一次，LLM 生成一次"的单向流水线。Agent 拥有 9 个专业工具，能自主规划检索策略——需要查什么、查几次、是否交叉验证、是否评估证据质量。这是一个会"思考"的检索系统。

**RAG 知识库**：最终交付的不是一个聊天机器人，而是一个可被检索、可被引用、可被审计的高质量知识库。每一条结论都精确追溯到原文的文献名、页码和证据等级。

**系统**：不是 demo，不是原型，不是"在理想条件下可以运行"的概念验证。是一个经过 110 项测试、40+ 缺陷修复、13 次 Git 迭代的真实可运行系统。

## 四、产品架构：一件事，五个步骤

MedRAG 将"从 PDF 到智能问答"这个复杂过程，拆解为五个清晰的步骤。每一步都对应一个明确的产品能力：

```
步骤 1: 上传        → 拖拽 PDF，系统自动接收、排队、分配资源
步骤 2: 解析        → VLM 逐页"阅读"，提取文字、表格、图表、版式信息
步骤 3: 语义切分    → 按 PICO 框架智能切分，保留完整的临床推理单元
步骤 4: 双引擎入库   → FAISS 向量索引 + LightRAG 知识图谱，双轨并行
步骤 5: Agent 问答   → 自然语言提问，Agent 多步推理，溯源回答
```

对用户而言，这五个步骤被压缩为一个体验：**上传 PDF，3 分钟后，你可以向它提问。**

### 4.1 步骤 2 的产品意义：VLM 解析

传统 PDF 解析器是"规则驱动"的——它们分析字符坐标、检测线条、匹配正则表达式。这在小范围可行，但在医学文献的复杂版式面前几乎必然失败。

MedRAG 选择了一个颠覆性的技术路径：**让 VLM（视觉语言模型）像人类一样"阅读"页面。** Qwen2-VL 1.2B 逐页将 PDF 渲染为图像，理解页面上的文字、表格和布局关系。双栏不是两个互相干扰的坐标带——它是模型自然理解的视觉结构。

**对用户的意义**：不需要担心"这篇论文是双栏排版的，解析器会不会搞混"。VLM 的视觉理解能力消除了这个担忧。

### 4.2 步骤 3 的产品意义：PICO 语义切分

传统 RAG 按固定字数切分文本——每 512 字一刀，像切豆腐一样把文献切成一堆均匀的小块。对于小说，这或许勉强可行。对于医学文献，这是灾难。

MedRAG 换了一个角度：**按照临床研究员做笔记的方式切分。** 一位优秀的研究员不会在"第 512 个字"处划一条线——她会在 Methods 结束、Primary Outcome 开始处做记号。

这就是我们的 PICO 11 类语义切分体系。它不是规则匹配，而是 LLM 理解内容后的语义分类。每个文本块都是一个完整的临床推理单元。

**对用户的意义**：当你检索"房颤抗凝方案"时，得到的不再是残缺的半句话，而是包含完整证据、出处、证据等级的语义段落。

### 4.3 步骤 4 的产品意义：双引擎架构

单引擎 RAG 有一个根本问题：**它只能回答"相似性"，不能回答"关联性"。**

用户问"房颤抗凝方案"，FAISS 能找到包含"房颤""抗凝""DOAC""华法林"的文本块。但它不知道"房颤是一种心律失常"、"DOAC 和华法林之间存在替代关系"——这些是概念之间的关系，而非文本之间的相似度。

LightRAG 知识图谱填补了这个空白。213 个节点和 25 条关系构成了一个医学概念网络：房颤 →（治疗关系）→ DOAC →（对比关系）→ 华法林。当用户提问时，这个网络提供的是**关联性知识**，而不仅仅是相似文本。

**对用户的意义**：检索结果不仅有"相关文献"，还有"这些文献之间的关系"。前者告诉你有什么，后者告诉你怎么理解。

### 4.5 步骤 5 的产品意义：Agent 多跳推理

绝大多数 RAG 系统是"一问一答"式的：用户提一个问题，系统检索一次，LLM 生成一个回答。对于"什么是房颤"这样的简单问题，这足够了。但对于"对比 Impella LP2.5 与 IABP 在急性心梗合并心源性休克中的疗效差异，并评估证据质量"——这需要跨文献对比、证据等级排序、一致性验证，单次检索如同用一个问题回答一本书。

Agent 的核心价值在于**自主决策**：它自己决定检索什么、检索几次、是否交叉验证、是否评估证据质量。9 个专业工具（search_rag、deep_retrieve、cross_check、estimate_grade、analyze_image 等）在 Function Calling 的驱动下被按需编排。

**对用户的意义**：用户不需要学习"如何正确地提问"。Agent 会自己搞清楚需要什么信息、去哪里找、找得够不够。

## 五、用户旅程

### 5.1 场景一：临床研究员的文献综述

王医生需要完成一篇关于"丙酸血症患者慢性肾脏并发症"的综述。

**传统方式**：下载 50+ 篇 PDF → 逐篇打开、阅读、标注 → 手动提取表格数据 → 整理引用列表 → 撰写综述。耗时：2-3 周。

**MedRAG 方式**：
1. 批量上传 50 篇 PDF（拖拽到页面，系统自动排队处理）
2. 等待处理完成（任务中心实时显示进度）
3. 在 Agent 问答中输入："汇总所有关于丙酸血症患者慢性肾脏病患病率和危险因素的文献，按证据等级排序，标注每篇的研究设计和样本量"
4. 获得结构化回答 + 完整引用列表
5. 点击每条引用，跳转到原文对应位置

**耗时**：上传 15 分钟 + 等待解析约 2 小时（后台自动）+ 提问 30 分钟。总计不到 3 小时——传统方式的 5%。

### 5.2 场景二：临床决策支持

急诊科刘医生收治了一名房颤合并脑卒中病史的 78 岁男性患者。出血风险评分 HAS-BLED=3，卒中风险评分 CHA₂DS₂-VASc=5。她需要确认：这种情况下的抗凝策略是什么？

**传统方式**：回忆指南内容 → 查阅科室文件 → 如果找不到，只能依靠经验判断。耗时：不确定（可能 10 分钟，也可能找不到答案）。

**MedRAG 方式**：
1. 打开浏览器，进入问答页面
2. 输入："高龄房颤患者，HAS-BLED=3，CHA₂DS₂-VASc=5，有脑卒中病史，抗凝方案选择？"
3. Agent 自动检索知识库中的相关指南和 RCT 证据
4. 返回建议 + 证据等级（GRADE 评级）+ 引用来源

**耗时**：1 分钟。

### 5.3 场景三：医疗大模型研发

某 AI 公司正在训练医疗对话模型。他们需要构建一个高质量的 RAG 知识库作为训练数据。

**传统方式**：爬取公开医学文献 → 用通用 PDF 工具解析 → 按固定字数切分 → 人工抽查质量（发现大量语义碎片）→ 重新标注 → 重新切分。整个流程耗时数月，且质量不可控。

**MedRAG 方式**：部署系统 → 批量导入 PDF → 自动完成 VLM 解析 + PICO 语义切分 + FAISS 索引 + LightRAG 图谱 → 直接调用 API 获取按 PICO 分类的高质量文本段。每个段落绑定文献名、页码和证据等级。

**价值**：数据准备时间减少 90%+，语义碎片化问题从根本上被消除。模型的幻觉率与知识库质量直接相关——高质量的知识库，是高质量医疗 AI 的前提。

## 六、竞争定位：我们有什么不同？

### 6.1 与通用 RAG 平台的对比

| 维度 | 通用 RAG 平台 | MedRAG |
|------|-------------|--------|
| PDF 解析 | 规则引擎 / OCR | VLM 逐页视觉理解 |
| 文本切分 | 固定 512 字滑动窗口 | PICO 11 类医学语义切分 |
| 检索 | 单一向量检索 | FAISS + LightRAG 双引擎 |
| 问答 | 一次检索 + 一次生成 | Agent 多步推理 + 回溯自纠 |
| 图表处理 | ❌ 不可检索的像素 | ✅ VLM 结构化数据提取 |
| 证据追溯 | 可能缺失 | ✅ doc · page · evidence_level |
| 知识图谱 | 通常无 | ✅ 213 节点医学概念网络 |
| 医学专业化 | 通用工具 | ✅ 循证医学 PICO/GRADE 体系 |

### 6.2 我们的护城河

**护城河一：VLM 解析的不可替代性。** 通用 PDF 解析器在处理中文医学文献的复杂版式时，其规则引擎在本质上受到限制——规则无法穷举所有版式变化。VLM 的视觉理解能力从根本上绕过了这个问题。这是一种范式级优势，而非渐进式改进。

**护城河二：PICO 语义切分的领域壁垒。** 设计一套 11 类医学语义分型体系，需要同时理解循证医学的方法论和 NLP 的技术约束。这是一个交叉学科的产物——纯 AI 团队不懂 PICO，纯临床团队不懂怎么让 LLM 做分类。

**护城河三：端到端的工程完整性。** 从 PDF 上传到 Agent 问答，不是 5 个独立模块的拼凑，而是一条被状态机精确管控的自动化流水线。每一步都有超时保护、错误处理、日志记录、数据一致性校验。这是 demo 和产品之间的本质区别。

## 八、用户反馈设计

产品设计中贯穿了一条原则：**让用户始终知道系统在做什么。**

- **上传进度实时可见**：任务中心的 6 态进度条（received → parsing → chunking → indexing → done），每 2 秒自动刷新
- **Agent 推理过程可视化**：右侧面板中，每一个工具调用、每一步耗时、每一次搜索的关键词都实时展示——用户看到的不是一个黑盒，而是一个透明的推理过程
- **引用一键溯源**：每条结论末尾的引用标签可点击展开，直接查看原文片段和出处信息
- **空状态引导**：每个页面在没有数据时显示的不是空白，而是清晰的引导文案和快捷操作入口
- **错误降级提示**：当 LLM 不可用时，系统不会崩溃或白屏，而是明确提示" AI 推理服务暂时不可用，以下为知识库直接检索结果"——诚实比完美更重要

## 九、为什么是现在？

2026 年的医疗 AI 领域正在经历一场深刻的范式转变。

第一代医疗 AI 是"知识库 + 检索"的简单组合。它的瓶颈在于知识库质量——固定字数切分导致语义碎片化，检索到的信息残缺不全。

第二代医疗 AI 引入了 LLM 进行生成增强。但 LLM 的展现能力反过来放大了知识库的缺陷——当 LLM 面对碎片化信息时，它会"脑补"缺失的部分。这就是**幻觉**。

第三代医疗 AI 的关键能力是**知识库质量 + 推理能力**的双重升级。这恰好是 MedRAG 的核心竞争力：PICO 语义切分解决知识库质量问题，Agent 多跳推理解决推理能力问题。两者缺一不可。

**我们正站在第三代医疗 AI 的起点上。** MedRAG 是我们对这个时代的回答。
